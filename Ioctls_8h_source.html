17 // Suffix strings for forming device names. Take the "device detail" path string
18 // returned by a SetupDiXxx enumeration and add on one of these, followed by
19 // a zero-based DECIMAL integer denoting the endpoint address. Use the resulting
20 // name string as the first argument in a call to CreateFile. Application
21 // code should use constructs like this to specify names in either Unicode or
22 // ANSI format: _T(MCHPUSB_PIPE_NAME). Driver code can use TEXT(MCHPUSB_PIPE_NAME), etc.
23 // For interrupt endpoints, using the "ASYNC" suffix means that you must use
24 // IOCTL_MCHPUPSB_WAIT_INTERRUPT to read input from the pipe. Using the EP
25 // suffix means you must use ReadFile or WriteFile, as appropriate to the endpoint's
26 // directionality.

32 // IOCTL definitions.
33 //
34 // Note that "input" and "output" refer to the parameter designations in calls to
35 // DeviceIoControl, which are the opposite of common sense from the perspective of
36 // an application making the calls.
37 //
38 // IOCTL_MCHPUSB_GET_VERSION -- get driver revision level
39 // Input:
40 // None
41 // Output:
42 // 32-bit revision level MMMMmmmm
43 //
44 // IOCTL_MCHPUSB_GET_DESCRIPTOR -- get descriptor
45 // Input:
46 // GET_DESCRIPTOR_PARAMETER structure (q.v.)
47 // Output:
48 // As much of specified descriptor as will fit in the buffer
49 //
50 // IOCTL_MCHPUSB_GET_CONFIGURATION_INFO -- get simplified information about configuration
51 // Input:
52 // None
53 // Output:
54 // DEVICE_INFO structure (q.v.)
55 //
56 // IOCTL_MCHPUSB_SET_CONFIGURATION -- configure device
57 // Input:
58 // SET_CONFIGURATION_PARAMETER structure (q.v.)
59 // Output:
60 // None.
61 // Note that this IOCTL will fail with STATUS_ACCESS_DENIED if any pipe handles are presently
62 // open for endpoints that are not in the new configuration.
63 //
64 // IOCTL_MCHPUSB_CONTROL_OUT -- issue control pipe command with output data
65 // Input:
66 // CONTROL_PARAMETER structure (q.v.)
67 // Output:
68 // Data to be sent to device (at most 4096 bytes)
69 // Note: you must use this IOCTL with care, so as not to put the device into a state other
70 // than expected by the driver. E.g., sending a SET_ADDRESS request would be a really bad idea.
71 //
72 // IOCTL_MCHPUSB_CONTROL_IN -- issue control pipe command with input data
73 // Input:
74 // CONTROL_PARAMETER structure (q.v.)
75 // Output:
76 // Buffer for data retrieved from device (at most 4096 bytes)
77 //
78 // IOCTL_MCHPUSB_WAIT_INTERRUPT -- await input from interrupt endpoint
79 // Input:
80 // None
81 // Output:
82 // Interrupt data provided by the matching input transaction on the endpoint
83
